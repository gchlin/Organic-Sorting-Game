# 設計決策清單（Design Decision Checklist）

> 用途：把「要做出專業設計需要想清楚的東西」全列出來，**一次只處理一桶**，談完就標 ✅ 並寫進 `README.md` 的決策日誌。
> 目的：避免繞路、節省討論成本。
> 建立：2026-05-11

---

## 這是什麼類型的遊戲

**教育型測驗遊戲（edutainment / quiz）＋ 街機元素（計時、HP、combo、對戰）。**
核心是 flashcard 練習（看結構式 → 辨識）包上「巫師 / 魔藥」的遊戲皮。同類參考：Kahoot、Duolingo 練習關、Wordle、brain-training。

設計上的隱含承諾：**好玩是手段，學會分類是目的。** 任何設計衝突，以「學會」為準。

## 決策清單（標註狀態：✅ 已定 / ⏳ 進行中 / ❌ 未做 / ❓ 未討論）

### §0 定位 Pillars ✅（2026-05-11 定案）
**正式 pillar**：《有機分類帽》是給**高中生**的有機官能基「快速辨識」練習遊戲——看結構式 → 分類；**電腦與手機都要版面友善**。成功標準依序：(1) 學生真的**學會分類**；(2) 學生覺得**有趣、願意主動一直練**。當「好玩」與「學會」衝突，以「學會」為準；但「無聊到沒人想玩」也算失敗。

**從 pillar 推出的判斷準則**（之後取捨的依據）：
- 「電腦＋手機都友善」→ 響應式是**核心承諾**（不是 nice-to-have），但仍排在收尾工序做。
- 「快速辨識」→ 單題節奏要快；**結構圖必須一眼看清**；答題流程不塞多餘步驟。
- 「願意一直練」→ 需要留存鉤子（圖鑑 / 累積 / 錯題重出 / 劇情），但**不能蓋過學習主軸**。
- 「學會 > 好玩（衝突時）」→ 例：Duel 不能變純手速；分類資料正確性優先於體驗。
- **視覺特效準則**（2026-05-11 追加）：**不做全螢幕亮暗閃光 / 高對比爆閃**（刺眼，對光敏感、易頭暈的學生不友善）。衝擊感用**色彩光暈 + 粒子 + 位移 + 輕微震動**表現。
- **占位準則**：所有還沒有美術的圖（角色、特效、徽章…）**一律先用 emoji 示意**，手感對了再決定是否畫真圖。

### §2 核心循環 + 三模式定位 ✅（2026-05-11 定案）
- 循環：看圖 → 選 → 回饋 → 下一題 ✅
- **三模式各司其職，只有 Duel 有「輸贏」**：
  | 模式 | 定位 | 結束條件 | HP / 計時 | 節奏 |
  |---|---|---|---|---|
  | **Practice 自我修煉** | **教** | 練完該關一輪題庫（洗牌過一遍）→ 結算「答對 X 題 / 答錯這幾題」，可再來一輪 | **無死亡**（HP 拿掉或改成「本輪 X/Y 題」進度條）；無倒數 | 慢、可細看；答錯有「為什麼」+ 分類帽教練 |
  | **Speed 競速挑戰** | **測** | 時間到（基礎 60 秒；答對 / 連對會**加回秒數** → 仍會結束） | **無死亡**（答錯只是浪費秒數）；倒數計時 | 快；連對 combo |
  | **Duel 巫師對決** | **玩 / 競技** | **先答對 N 題（暫定 8）獲勝** | 無倒數；每題前 **~1.5 秒閱讀期**（按鈕鎖住）；答錯鎖 2 秒；HP 條 → 改當「到 N 題」的進度視覺 + 打架特效 | 中快 |
- **精神**：把「答錯扣血到爆炸」這個對新手不友善的機制，從學習用的 Practice / Speed 拿掉，只留在競技的 Duel（→ 順帶解掉 §6 一半的問題）。
- **Duel 公平性對策（定案）**：**各自題目流 + 閱讀期**。
  - 兩名玩家**各有自己的題目**（從該關洗牌佇列各抽各的）；答對 → **不等對方**，自己往下一題；先答對 N 題者勝（一場「賽跑」）。
  - 每題開始有 **~1.5 秒閱讀期**（按鈕鎖住、顯示「準備…」）→ 之後開放作答。閱讀期的作用：強迫「先看懂結構式」，杜絕不看就亂按；也讓「純反射」的上限被壓住（每題至少 1.5 秒）。**我們不是單純比反應。**
  - 答錯 → 該玩家鎖 2 秒（懲罰，比閱讀期長）。
  - ⚠️ **連帶影響**：「各自題目」推翻 §8 的「桌面 Duel 共用一組選項」。桌面 Duel 改成「**左右兩塊，各自題目 + 各自 2x2 選項**」（不旋轉）；鍵位 P1=`A/F/Z/C` 配左格、P2=`4/6/1/3` 配右格 仍適用。手機 Duel 維持上下分割鏡像。→ 已記成「給 Codex 的修正」（見 §8）。
- [ ] Speed「答對加秒數」的具體公式（草案：答對 +2 秒；combo 2 = +3、combo 3+ = +4；可設或不設上限）
- [ ] Duel 的 N（暫定 8）、閱讀期秒數（暫定 1.5）、答錯鎖秒數（暫定 2）— 之後 playtest 微調

### §5 動機 / 留存 ✅（2026-05-11 定案；架構待建，純前端 localStorage）
**要做的留存鉤子**：
1. **進度可見**：「已認得 X / Y 種分子」、各 Level 完成度。
2. **錯題系統**：答錯的題記下來 → 下輪 / 練習時優先重考 → 攻克後標 ✓。（同時是學習工具）
3. **個人累積計數**：累計**答對題數**（不是最高分、不是正確率，就是「總共答對幾題」這種一路往上加的成就感）→ 達門檻**解鎖勳章**。
   - Duel 模式**不計入**個人累積（避免「兩人共機誰的分」歸屬問題）；只算 Practice / Speed。
4. **圖鑑**（三類，各自獨立收集）：
   - (a) 分類圖鑑：解鎖認得過的化合物 + 它的官能基分類
   - (b) 性質 / 用途圖鑑：解鎖該化合物的性質、用途小知識
   - (c) 化學迷因梗圖圖鑑（梗圖素材使用者另尋；localStorage 只存「解鎖編號」，圖檔放 `assets/`）
- **暫不做**：連續天數 streak（壓力）、最高分排行榜、複雜成就樹。
- 內在動機（精熟感：進度 / 錯題攻克）優先於外在獎勵（勳章 / 圖鑑 / 梗圖）。

### §17 存檔策略（localStorage / 純前端）⏳（2026-05-11，方向已定）
- 純前端：`localStorage` 存一份 JSON。**做匯出 / 匯入存檔**（明碼 JSON 字串，**先不加密**——學生想抄改也無所謂，這只是練習進度）。需要記錄的參數都寫進去沒關係。
- **限制**：每台機器 / 每瀏覽器一份；清快取就沒；不跨裝置同步；容量 ~5–10MB（純文字夠用，**不要存圖片 base64**）。
- **多人共機（教室電腦）**：Duel 不計入個人累積（已決定）；單人累積算「這台機器」的——匯出存檔可讓學生帶著走。是否做「玩家名字切換」之後再說（先單一存檔）。
- 存檔結構草案：
  ```json
  {
    "version": 1,
    "totalCorrect": 0,                          // 累積答對題數（非比率非最佳；解鎖勳章用）
    "badges": [],                               // 已解鎖勳章 id
    "seenMolecules": [],                        // 圖鑑·分類：已認得的分子 key
    "knownProperties": [],                      // 圖鑑·性質用途：已解鎖的分子 key
    "memes": [],                                // 圖鑑·迷因梗圖：已解鎖編號
    "wrongQueue": {},                           // 錯題：{ molKey: 還沒攻克的次數 }
    "storyUnlocked": [],                        // 已解鎖劇情的關卡 ["level1", …]
    "levelClears": {},                          // 各關是否達門檻過 { "level1": true, … }
    "bestScores": {}                            // 可選：{ "speed_level3": 120, "duel_level6": 80, … }
  }
  ```
